# Studio feature request: honor `AGENTS.md` on GitHub import

> **From**: the Tama team
> **To**: deco studio (mesh) maintainers
> **Status**: spec / request for comment

## Summary

When the Studio coding-agent imports a GitHub repo, look for `AGENTS.md`
at the repo root and treat it as setup instructions. If present, the
agent reads it on first conversation and follows the customize → commit
→ deploy lifecycle described there.

## Why

Template repos like Tama are "this code is a fork point, not a finished
product" — they intentionally ship as placeholders that an agent fills
in. Without a convention for *where the agent finds those instructions*,
every template needs to invent its own onboarding story.

`AGENTS.md` is already an emerging informal convention across the
agentic-IDE ecosystem (Cursor, Aider, Claude Code). Standardizing on it
in Studio means template authors write *one* file and get a coherent
onboarding inside Studio without per-IDE forks.

## What we want

When a user clicks "Import from GitHub" in Studio:

1. Studio clones the repo into a workspace.
2. The coding-agent reads `AGENTS.md` if it exists at the repo root.
3. On first user message (or via an "Set up this template" CTA), the
   agent treats `AGENTS.md`'s lifecycle as the active conversation
   script — asking the questions, making the edits, committing.
4. After the lifecycle is complete, the agent stops treating it as
   load-bearing and reads `CLAUDE.md` for ongoing development guidance.

## Proposed mechanics

### Detection

On clone, the orchestrator checks for `AGENTS.md` at:

1. `/AGENTS.md` (root, canonical location).
2. `/.github/AGENTS.md` (some projects prefer this).

If both, the root file wins.

### First-contact UX

The first message to the user is generated by the coding-agent and
references the file. Example:

> Hey! I see `AGENTS.md` in this repo — looks like it's a template that
> needs to be adopted for a specific [pet/garden/project]. Want me to
> walk you through it?

If the user says yes, the agent follows `AGENTS.md`'s lifecycle. If they
say no, the agent backs off and waits for direction.

### Lifecycle phases

`AGENTS.md` files conventionally have these phases (Tama uses this
shape):

1. **CUSTOMIZE** — gather inputs through conversation, edit config files.
2. **COMMIT** — git commit with a deterministic message.
3. **DEPLOY** — run provisioning + deploy commands.
4. **OPERATE** — register the deployed MCP endpoint as a Studio agent
   so the same chat becomes admin.

Studio can offer affordances at each phase boundary (e.g. a "Deploy"
button after the commit lands).

### Completion detection

The agent should mark `AGENTS.md` as "done" after step 4. Two options:

1. **State key in workspace** — a workspace metadata field like
   `template_adopted_at` that Studio sets when the agent reports
   completion.
2. **Convention in the file itself** — the bottom of `AGENTS.md` says
   "after step 4 this file stops being load-bearing." The agent simply
   stops reading it once the operate phase is complete.

Option 2 is lighter-weight and what Tama assumes today.

## What this is not

- Not a replacement for `CLAUDE.md`. `CLAUDE.md` is for ongoing dev
  guidance; `AGENTS.md` is specifically the first-import setup protocol.
- Not a config file. `AGENTS.md` is human-readable markdown the agent
  parses with its general reading skill, not a JSON schema.
- Not a way to ship code. Anything the agent does is via its normal
  file-edit / shell tools — `AGENTS.md` just *describes* what to do.

## Reference: Tama's AGENTS.md

See `/AGENTS.md` in the Tama repo for the canonical structure we'd be
asking Studio to honor. Key sections:

- "What this repo is" — orienting text.
- "Lifecycle" with numbered phases.
- "What you should not do" — guardrails.

The file is ~150 lines, plain prose with shell snippets where needed.

## Acceptance

When this lands, the Tama flow looks like:

1. Owner goes to studio.decocms.com.
2. Clicks "Import from GitHub", pastes `https://github.com/deco-cx/tama`.
3. Studio's coding-agent says: *"Looks like this is a Tama template.
   Want me to set it up for your pet?"*
4. Owner says yes; agent walks them through the conversation, edits
   files, generates the sprite, commits, deploys.
5. End state: the human has a deployed `tama-<theirpet>.workers.dev`
   *and* a Studio agent registered to its MCP endpoint, ready to chat.

The entire flow happens inside Studio. The human never opens a terminal.
